## 1. 불변 설계 원칙 (Invariants)

아래 7개는 **구현 전체에서 위반 불가**한 원칙이다. 코드 리뷰·테스트의 1차 기준이며, 각 원칙은 규제 보강안의 특정 축을 코드로 강제한 것이다.

| # | 원칙 | 구현 강제 지점 | 근거(보강) |
|---|------|--------------|-----------|
| **P1** | **한도는 기계 산식** — `limit = settlement_expected × (1 − haircut)`. AI 모델 출력은 한도 계산식에 **입력되지 않는다** | `rule_engine/limits.py` 단일 구현. agents 모듈은 한도 필드를 출력 스키마에 갖지 않음 | 보강2 |
| **P2** | **자격은 네거티브 스크리닝** — 폐업·압류·중복양도·결격 사유만 거른다. 점수 기반 자격 게이팅 금지 | `rule_engine/screening.py`. 스크리닝 결과는 `pass / fail(reason)` 이진 | 보강2 |
| **P3** | **회수확률·위험점수 = 라우팅 신호** — 자동집행/은행원검토 **분기에만** 사용. AI는 거절을 결정하지 않는다(결격 제외 거절·보류는 인간만) | `rule_engine/routing.py`. 라우팅 결과에 `REJECT` 없음 — `AUTO_EXECUTE / BANKER_REVIEW / INELIGIBLE` 3종뿐 | 보강2·§37의2 |
| **P4** | **결정론/LLM 분리** — 판단·집행 경로에 LLM 부재. LLM은 FACT_BLOCK 기반 설명·RAG·대화 전용 | `llm/` 모듈은 `rule_engine/`·`agents/`를 import 할 수 없음(단방향 의존). validator 하드 리젝트 | 명세서 ② |
| **P5** | **haircut ≠ 이자** — haircut은 채권평가감(환불·취소율)만. 기간 수수료(이자)는 별도 계산·별도 고지, **연환산 ≤ 법정상한(20%) assert** | `rule_engine/fees.py`. `assert annualized_rate <= LEGAL_CAP` 위반 시 집행 자체가 불가 | 보강3 |
| **P6** | **전 결정 감사 가능** — 모든 결정에 audit record + §37의2 대응 인터페이스(설명요구·인적재심) | `audit/` 모듈. `RuleEngineDecision`은 `audit_id` 없이 생성 불가 | 보강4·§37의2 |
| **P7** | **수익성 에이전트는 개별 건에 관여하지 않는다** — 정책 파라미터(수수료율·자동집행 임계값·포트폴리오 노출) **제안만**, 적용은 인간 승인 후 | `agents/profitability.py` 출력은 `PolicyProposal`(건 단위 필드 없음). 오케스트레이터가 개별 건 평가 경로에 전달하지 않음 | 보강2·§37의2 |

> P7 해설 — 기능명세서의 "수익성 에이전트: 한도·금리 최적화"의 구현 해석: **개별 건의 한도·금리를 정하는 것이 아니라**, 규칙엔진이 쓰는 정책 파라미터 테이블을 포트폴리오 데이터로 최적화해 *제안*하는 역할이다. 제안 → 위험 에이전트 동의 검증 → 인간(은행) 승인 → 테이블 반영. 이렇게 해야 "사전승인 정책 내 자동 집행" 구조와 §37의2 방어가 유지된다.

## 2. 전체 아키텍처

```

 [트리거 3종]          │              ORCHESTRATOR                   │

  · 정산예정 발생      ─▶│            → ROUTE → ACT → VERIFY → LEARN  │
  · 은행원 조회        │   (동적 전이: 재산정 요청·동의 요건·재시도)      │
                      └──────┬──────────────────────────┬───────────┘
                             │ 호출(병렬)                 │ 종합 리포트
              ┌──────────────┼──────────────┐            ▼
        ┌─────┴─────┐ ┌──────┴────┐ ┌───────┴───┐ ┌─────────────────────┐
        │ 회수확률    │ │ 위험       │ │ 정책       │ │   RULE ENGINE        │
        │ agent     │ │ agent     │ │ agent     │ │  (결정론 최종 중재)     │

        │ +SHAP     │ │ 카드깡 룰   │ │           │ │ ②한도=기계 산식       │
        └───────────┘ └───────────┘ └───────────┘ │ ③수수료 분리+법정캡    │
        ┌───────────┐                             │ ④3단계 라우팅         │
        │ 수익성     │──(정책 제안·인간 승인 큐)──────▶│ ⑤충돌 중재·감사       │
        │ agent     │   ※ 개별 건 경로에 미참여       └──────────┬──────────┘
        └───────────┘                                          │ 결정+근거
                      ┌────────────────────────────────────────┼───────────┐
                      │ LLM EXPLAINER (Gemma 12B·vLLM·온프레미스) │           │
                      │  FACT_BLOCK 주입 → 자연어 설명 → validator │◀──────────┘
                      │  정책 RAG(한국어 임베딩+pgvector) · 대화     │  숫자만 전달
                      └──────────────────────┬───────────────────┘
                                             ▼
       [행동 계층]  FastAPI ──▶ React 은행원 코파일럿 UI(frontend/) · 위험 알림
                  · 자동 집행(시뮬) · 정산일 자동상계(시뮬) · 거절-연착륙 호출
                                             ▼
       [저장 계층]  합성 정산 DB · 정책 코퍼스(pgvector) · 모델 레지스트리 · 감사 로그
```

**의존 방향(강제):** `llm → (없음)` / `agents → data` / `rule_engine → data` / `orchestrator → agents, rule_engine, llm, audit` / `api → orchestrator`. LLM·agents가 rule_engine을 호출하는 역방향 의존 금지(P4).

## 3. 데이터 계약 (JSON 스키마)

모든 에이전트는 **구조화 출력**으로만 보고한다(자유 텍스트 금지). pydantic 모델로 구현(`data/schemas.py`).

### 3-1. 공통 봉투 — `AgentReport`

```jsonc
{
  "agent": "recovery | risk | policy",        // profitability는 별도(3-6)
  "agent_version": "recovery-1.0.0",
  "run_id": "run_20260610_0001",              // 오케스트레이터 발급, 전 보고 공통
  "merchant_id": "M-0001",
  "produced_at": "2026-06-10T02:00:00+09:00",
  "latency_ms": 142,
  "payload": { /* 에이전트별 스키마 (아래) */ },
  "evidence": [ /* SHAP 항목 또는 룰 히트 — 설명·감사용 */ ]
}
```

## 4. 오케스트레이터 (`orchestrator/`)

### 4-1. 트리거 3종

| 트리거 | 구현 | 대상 |
|---|---|---|
| 매일 밤 능동 스캔 | 스케줄러(MVP: APScheduler / 데모: "스캔 실행" 버튼) | 관할 전 소상공인 |
| 정산예정 발생 | 합성 데이터 생성기의 이벤트 큐 | 해당 가맹점 |
| 은행원 조회 | `GET /api/merchants/{id}/assessment` 호출 시 on-demand | 해당 가맹점 |

### 4-2. 상태머신 (건 단위 평가 런)

```
COLLECT ──▶ ASSESS ──▶ MEDIATE ──▶ ROUTE ──▶ ACT ──▶ VERIFY ──▶ LEARN
   │           │           │          │         │        │          │
 스냅샷      병렬 호출:     규칙엔진    3단계     집행/     정산일      F6 피드백
 적재     recovery+risk   decide()   분기     에스컬/    자동상계     (본선)
          (+policy는      ↑          │       연착륙     모니터
           라우팅 결과     │ 동적 전이  │
           따라 lazy)     │          │
                    [risk flags?]───┘
                     YES → recovery 재산정 요청(1회) 후 MEDIATE 재진입
```

**상태별 계약:**

- `COLLECT`: `MerchantSnapshot` 생성. `settlement.confirmed == false`면 즉시 종료(평가 대상 아님).
- `ASSESS`: `recovery`·`risk` **병렬** 호출(asyncio.gather). `policy`는 ①능동 스캔 모드에서는 항상, ②건 평가에서는 라우팅 결과가 `BANKER_REVIEW`/`INELIGIBLE`일 때 연착륙용으로 lazy 호출.
- `MEDIATE`: 규칙엔진 `decide(snapshot, reports)` 1회 호출. **여기서만 한도·수수료·라우팅이 결정**된다.
- `ROUTE`: 결정에 따라 분기 — `AUTO_EXECUTE`(ACT로), `BANKER_REVIEW`(은행원 큐 적재+알림), `INELIGIBLE`(연착륙 패키지 생성).
- `ACT`: 집행 기록 생성(MVP: 시뮬 원장), 거절·보류 분기에서 F2 정책·F3 연착륙 자동 호출.
- `VERIFY`: 정산일 자동상계 시뮬 → 회수 결과 기록(F5 `[시뮬]`).
- `LEARN`: 회수 결과로 모델·임계값 재보정 큐 적재(F6 `[설계]` — 인터페이스만 구현).

### 4-3. 동적 전이 (자율성 — 정적 워크플로우가 아님의 증거)

기능명세서 v2의 "자율성 증거"를 다음 3개 메커니즘으로 구현한다:

1. **재산정 요청:** `RiskReport.request_recovery_recompute == true` → 오케스트레이터가 회수확률 에이전트를 위험 플래그 컨텍스트와 함께 재호출(`recompute_of` 체인). **최대 1회**(무한 루프 방지). 재산정 후 MEDIATE 재진입.
2. **동의 요건:** 수익성 에이전트의 `PolicyProposal`은 위험 에이전트의 동의 검증(`risk_agent_consent`: 제안 파라미터로 백테스트 시 위험 한계 내인지)을 통과해야 인간 승인 큐에 올라간다. 거절되면 사유와 함께 반려.
3. **동적 임계값:** `auto_exec_threshold`는 코드 상수가 아니라 **정책 테이블 값**(`rule_engine/tables.py` → MVP는 JSON 파일, 버전 태깅). F6 루프·인간 승인으로만 변경된다.

### 4-4. 실패 처리 (fail-safe 방향 고정)

| 실패 | 처리 | 원칙 |
|---|---|---|
| 에이전트 타임아웃·예외 | 해당 보고를 "결측"으로 두고 **`BANKER_REVIEW`로 강등** (자동집행 금지) | 보수 방향 — 자동으로 돈이 나가는 경로는 완전한 정보에서만 |
| 규칙엔진 cap assert 실패 | 런 전체 중단 + 감사 로그 + 관리자 알림 | P5는 협상 불가 |
| LLM validator 리젝트 | 재생성 최대 2회 → 실패 시 **결정론 템플릿 문장으로 폴백** (수치 끼워넣기) | 설명 실패가 집행을 막지 않되, 거짓 설명은 절대 노출 안 됨 |
| LLM 서버 다운 | 템플릿 폴백 모드로 전체 전환 | LLM은 비필수 경로(P4) |

## 5. 전문 에이전트 명세 (`agents/`)

공통: `base.py`의 `SpecialistAgent` 추상 클래스(`run(snapshot, context) -> AgentReport`) 구현. 각 에이전트는 자기 모델·도구만 알고, 다른 에이전트·규칙엔진을 호출하지 않는다(조율은 오케스트레이터만).

### 5-1. 회수확률 에이전트 (`recovery.py`)

| 항목 | 내용 |
|---|---|
| 책임 | 정산 채권의 회수 가능성 추정 → **라우팅 신호** 제공 |
| 입력 | `MerchantSnapshot` + 시계열 피처(반복매출·정산이력·상권·시즌) |
| 출력 | `RecoveryReport` (3-3) — `recovery_prob` + SHAP top-k |
| 모델 | LightGBM (이진: 정상회수 여부) + SHAP TreeExplainer. MVP는 합성 데이터 학습 |
| 학습 데이터 주의 | 운영대행 정산은 회수율 ~100% → 음성 라벨 부족. **합성 음성 케이스(환불 폭증·이상거래) 증강**으로 보완 (v2 명세서 "모델 학습 데이터" 절) |
| **하지 않는 것** | 한도 산정(P1) · 자격 판정(P2) · 거절 결정(P3) |

### 5-2. 위험 에이전트 (`risk.py`)

| 항목 | 내용 |
|---|---|
| 책임 | 개인별 이상 패턴 조기 감지(연체 전 골든타임) + 모럴해저드 적출 |
| 입력 | 정산·매출 시계열 |
| 출력 | `RiskReport` (3-4) — 플래그·심각도·카드깡 룰 히트 |
| 모델·도구 | scikit-learn 이상탐지(IsolationForest 기본) + 결정론 룰셋(단기 매출 급증, 동일 단말·시간대 환불 burst 등) |
| 자율 행동 | 이상 신호 시 `request_recovery_recompute` 발화 → 회수확률 재산정 유도 |
| **하지 않는 것** | 거절 결정 — 플래그는 **은행원 검토 라우팅**으로만 이어진다(P3). 플래그 발견 시 F2·F3 연착륙이 자동 동반 |

### 5-3. 정책 에이전트 (`policy.py`)

| 항목 | 내용 |
|---|---|
| 책임 | 정책자금·지원금 매칭(F2) — 능동 매칭 + 거절-연착륙 대안 |
| 입력 | 소상공인 프로필 + 정책 공고 코퍼스 |
| 출력 | `PolicyReport` (3-5) — 매칭 정책·자격 근거·신청 동선·RAG 근거 청크 |
| 도구 | 정책 RAG(한국어 임베딩 + pgvector) → **자격 룰 검증(결정론)** 2단 구성. RAG가 후보를 찾고 룰이 자격을 확정 — 환각 정책 추천 차단 |
| **하지 않는 것** | 자격 룰 통과 못 한 정책을 "추천"으로 노출(후보까지만) |

### 5-4. 수익성 에이전트 (`profitability.py`) — P7 적용

| 항목 | 내용 |
|---|---|
| 책임 | **포트폴리오 레벨** 정책 파라미터 최적화 제안: 수수료율(≤법정캡)·자동집행 임계값·노출 한도·haircut 테이블 보정 |
| 입력 | 회수 실적 집계·포트폴리오 노출·운영 비용 가정 |
| 출력 | `PolicyProposal` (3-6) — `merchant_id` 필드가 **스키마에 존재하지 않음** |
| 적용 절차 | 제안 → 위험 에이전트 동의 검증 → **인간 승인** → `tables.py` 정책 테이블 버전 갱신 |
| **하지 않는 것** | 개별 건 한도·금리 결정, 개별 건 평가 런 참여 |
| MVP 범위 | `[설계]` — 스키마·승인 큐 인터페이스만 구현, 최적화 로직은 본선 |

## 6. 규칙엔진 (`rule_engine/`) — 결정론 최종 중재

```python
# rule_engine/engine.py — 의사코드 (실제 구현 기준)
LEGAL_CAP = 0.20                      # 이자제한법·대부업법 연 20% (2026 기준)

def decide(snapshot: MerchantSnapshot,
           recovery: RecoveryReport | None,
           risk: RiskReport | None,
           policy_version: str) -> RuleEngineDecision:

    audit = audit_log.open(snapshot.merchant_id)          # P6 — 가장 먼저

    # ① 네거티브 스크리닝 (P2) — 점수 무관, 사실 기반 이진
    s = screening.check(snapshot.screening_facts)         # 폐업·압류·중복양도·3종동의
    if not s.passed:
        return Decision(INELIGIBLE, reasons=s.fail_reasons,
                        soft_landing=soft_landing_package(snapshot), audit=audit)

    # ② 한도 — 기계 산식 (P1). AI 출력은 이 블록에 등장하지 않는다.
    h = tables.haircut_for(snapshot.sector, snapshot.refund_stats)   # 분포 꼬리·연간 누적 반영
    limit = floor(snapshot.settlement.expected_amount * (1 - h.rate))

    # ③ 기간 수수료 — haircut과 분리 (P5)
    fee = fees.calc(limit, tables.fee_annual_rate(policy_version),
                    days_until(snapshot.settlement.settlement_date))
    assert fee.annualized <= LEGAL_CAP                    # 위반 시 집행 불가(예외 전파)

    # ④ 3단계 라우팅 (P3) — 신호는 분기에만 쓴다
    if risk is None or recovery is None:                  # 결측 = 보수 강등 (4-4)
        return Decision(BANKER_REVIEW, limit, fee, reasons=["agent_report_missing"], audit=audit)
    if risk.risk_flags:
        return Decision(BANKER_REVIEW, limit, fee,
                        reasons=flag_reasons(risk),
                        soft_landing=soft_landing_package(snapshot), audit=audit)
    if recovery.recovery_prob < tables.auto_exec_threshold(policy_version):
        return Decision(BANKER_REVIEW, limit, fee,
                        reasons=[f"recovery_prob {recovery.recovery_prob} < threshold"], audit=audit)

    return Decision(AUTO_EXECUTE, limit, fee,
                    reasons=["screening:pass", "risk_flags:0", "prob≥threshold"], audit=audit)
```

**정책 테이블 (`tables.py`):** haircut 테이블(업종 × 환불률 p95 구간 → haircut율, 연간 누적 한도 차감), `fee_annual_rate`, `auto_exec_threshold`, 노출 한도. **MVP는 버전 태깅된 JSON 파일** — 모든 `RuleEngineDecision`이 `policy_snapshot_version`을 기록해 "어느 사전승인 정책으로 결정했는가"를 감사 가능하게 한다. 변경은 수익성 에이전트 제안 → 인간 승인 경로만(4-3).

**충돌 중재 규칙:** 보고 간 모순(예: 회수확률 높음 + 위험 플래그 존재) 시 **항상 보수 우선** — 위험 플래그가 라우팅을 지배한다. 순서: `INELIGIBLE > BANKER_REVIEW > AUTO_EXECUTE`.

## 7. LLM 설명 계층 (`llm/`)

| 구성 | 내용 |
|---|---|
| 모델·서빙 | 온프레미스 Gemma 12B 계열, 4bit 양자화, vLLM. **MVP 폴백:** 로컬 GPU 미가용 시 결정론 템플릿 모드(`llm/templates.py`)로 동작 — 데모가 LLM 없이도 완주 가능해야 함 |

| validator (`validator.py`) | 생성문에서 수치·비율·금액을 정규식 추출 → FactBlock 수치와 **전수 대조**. 불일치·없는 숫자 발견 시 하드 리젝트 → 재생성(≤2회) → 템플릿 폴백. 라우팅 결과와 모순되는 단정("자동 승인되었습니다" vs `BANKER_REVIEW`)도 리젝트 패턴에 포함 |
| 정책 RAG (`rag.py`) | 한국어 임베딩 + pgvector(**MVP 폴백:** FAISS in-memory). 정책 에이전트의 후보 검색 + 대화형 질의 응답. 응답은 `source_chunk_ids` 필수 |

## 8. Agent Loop ↔ 기능(F1~F6) 매핑과 시나리오 트레이스

| 기능 | 모듈 경로 | MVP |
|---|---|---|
| F1 선지급(한도+라우팅) | `rule_engine/limits.py·routing.py` + `agents/recovery.py` | `[구현]` |
| F2 정책 매칭 | `agents/policy.py` + `llm/rag.py` | `[구현]` 샘플 정책 |

**트레이스 ① 봄(자동집행):** 정산예정 발생 → COLLECT(스냅샷, 3종 동의 OK) → ASSESS(recovery 0.96 / risk 플래그 0) → MEDIATE(스크리닝 pass → 한도 300만×0.92=276만 → 수수료 9,800원 cap PASS → `AUTO_EXECUTE`) → ACT(시뮬 집행+은행원 알림) → VERIFY(D+13 자동상계) → LEARN(결과 적재).

**트레이스 ② 가을(검토→보류→연착륙):** 야간 스캔 → risk가 `SALES_DROP_3W` HIGH 플래그 + 재산정 요청 → recovery 재산정(0.96→0.61) → MEDIATE(`BANKER_REVIEW` — **거절 아님**) → policy lazy 호출(연착륙 패키지: 정책자금 F2·상환유예·재신청 안내) → 은행원 화면에 SignalCard(검토)+대안 동시 표시 → 인간이 보류 결정 → audit 기록.

- **기술 스택:** Python 3.12 · FastAPI · pydantic v2 · LightGBM · SHAP · scikit-learn · (본선) vLLM+Gemma 12B 4bit · pgvector/FAISS · 한국어 임베딩 · React(기존 frontend/) · APScheduler.
- **용어:** *라우팅 신호* — 자동집행/검토 분기에만 쓰이는 AI 출력(한도·자격과 무관) / *정책 테이블* — 인간이 사전 승인한 파라미터 집합(버전 태깅) / *FactBlock* — LLM에 주입되는 결정론 수치 봉투 / *연착륙 패키지* — 거절·보류 시 자동 생성되는 정책매칭+유예+재신청 대안.
- **이 문서가 다루지 않는 것:** 실데이터 계약·운영대행 연동(Phase 1), 혁신금융서비스 신청 절차, 본선 발표 자료 구성.
